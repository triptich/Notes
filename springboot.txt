Transactions

A transaction is a sequence of operations executed as a single unit of work.
If all operations succeed then Commit
If any operation fails then Rollback
This prevents inconsistent data

For eg : Decbit Account A + Credit Account B
If credit fails -> debit must also rollback

ACID Properties
Spring transactions are based on ACID gurantees
Atomicity : All nor nothing
Consistency : DB moves from one valid state to another
Isolation : Transactions dont interfere with each other
Durability : Committed data is permanent

Transaction Management in Spring Boot
Spring Boot provides declarative transaction management using :
@Transactional annotation

What Spring does internally :
Opens transaction before method execution
Executes business logic
  Commits if success
  Rolls back if exception occurs

@Service
public class OrderService{
 @Autowired
private orderRepository orderRepository;

@Transactional
public void createOrder(Order order){
    orderRepository.save(order);
  }
}

Propagation :
Defines how transaction behaves when another transactional method is called
Types :
REQUIRED(default) : Join existing transaction or create new one
@Transactional(propagation = Propagation.REQUIRED)

REQUIRES_NEW : Always start new transaction. Suspends outer transaction. Used in Audit logging
@Transactional(propagation = Propagation.REQUIRES_NEW)

SUPPORTS :  USes existing transaction if present, else runs without tansaction
@Transactional(propagation = Propagation.SUPPORTS)

NOT_SUPPORTED : Suspends current transaction. Runs non-transactional
@Transactional(propagation = Propagation.NOT_SUPPORTED)

MANDATORY : Must have existing transaction. Throws exception if none exists
@Transactional(propagation = Propagation.MANDATORY)

NEVER : Fail if transaction exists
@Transactional(propagation = Propagation.NEVER)

NESTED : Nested transaction with savepoint. partial rollback possible
@Transactional(propagation = Propagation.NESTED)

ISOLATION LEVELS
Defines how transactions interact with each other 

DEFAULT : Uses DB default isolation level
READ_UNCOMMITTED : Nothing(dirty reads allowed)
READ_COMMITTED : Prevent dirty reads
REPEATABLE_READ : Prevent dirty + non-repeatable
SERIALIZABLE : Fully isolated

Rollback Rules :
By default
Rollabck happens if there is Run Time Exception
No rollback if there is checked exception

@Transactional(rollbackFor = Exception.class)

How @Transactional Works Internally
Spring used AOP (proxy-based mechanism)
Flow :
Proxy intercepts method call
Starts transaction
Executes method
Commit/rollback

Where to use Transactions
Service Layer( best practice)
Avoid in controller layer
Use in business logic methods

@Transactional doesnt work on private methods


Pagination :
Pagination is splitting of large datasets into smaller chunks instead of returning everything at once.
Why is it Required :
Reduces memory usage on server
Improves API Response time
Reduces network bandwidth
Better UX

Pageable:
Interface representing pagination info(page,size,sort)
Passed from Controller to repository

Pageable pageable = PageRequest.of(page,size);

Page:
Represents paginated result
Contains both data + metadata

Methods are :
page.getContent();
page.getTotalEmements();
page.getTotalPages();
page.getNumber();
page.isLast();

PageRequest
Concrete implementation of Pageable
PageRequest.of(page,size);
PageRequest.of(page,size,Sort.by("name").ascending());

Layer Wise Implementation
Entity

@Entity
public class USer{
@Id
@GeneratedValue

}

public interface UserRepository extends JPARepository<User,Long>{

Types of Pagination
1) Offset based Pagination(Default in Spring Boot)
Uses page and size
Slow for large data and can jump to any page
Internally uses:

2) Cursor Pagination
Uses last record as reference(eg ID or timestamp)
Fast for large datasets.
Used in real time systems(feeds,chat,logs)

Locking in Spring Boot
Locking in Spring Boot is used to handle concurrent data access and maintain data consistency when multiple transactions try to read/update the same data

Why Locking is Needed
In concurrent systems :
Multiple users/threads access same DB record
Leads to isisues :
  Lost Update
  Dirty read
  Non-repeatable Read
  Race conditions

Types of Locking
1) Optimistic Locking
2) Pessimistic Locking

1) Optimistic Locking : Assumes conflicts are reare
Does not lock DB rows
Instead uses a version field (@Version) to detect conflicts
Fails transaction if another update already happened

How It Works :
Read Entity -> store version
Update entity -> compare version
If version changed -> throws Exception

@Entity
public class Product{
@Version
private Long version;
}

User A reads version = 1
User B updates - version becomes 2
User A tried update -> fails with OptimisticLockException

Advantages
No DB Locking - High Performance
Good for read-heavy systems

Disadvantages
Needs retry logic
Not suitable when conflicts are frequent

2) Pessismistic Locking :
Assumes conflicts are likely
Locks the DB row immediately
Other tranactions must wait

Types (LockModeType)
PESSIMISTIC_READ : No updates allowed
PESSIMISTIC_WRITE : No read/write allowed

Implementation
@Repository
public inerface ProductRepository extends JPARepository<Product,Long>{

  @Lock(LockModeType.PESSIMISTIC_WRITE)
  @Query("Select p from product p where p.id =:id")
Optional<Product> findByIdForUpdate(Long id);
}

What happens
Transaction A fetches Records - Lock acquired
Transaction B tries - waits
A completes then B Proceeds

Advantages:
Strong consistency
Prevents lost updates completely

Disadvantages :
Performance overhead
Can cause deadlocks

Locking works only inside @Transactional
Optimistic uses @Version
Pessimistic uses @Lock and LockModeType
Without locking - leads to lost update problem
Optimistic - Non-Blocking
Pessimistic - Blocking

Use Optimistic :
Profile update
Analytics
Content management

Use Pessimistic
Banking transactions
Seat Booking
Stock/Inventory

JWT Working
JWT(JSON Web Token) is a compact, self contained token used for authentication and authorization
Stateless - no session stored on server
Token itself contains user info(claims)
Used in REST APIs , microservices 
It is signed so server can verify it without database lookup

Structure of JWT
JWT has 3 parts
Header
Payload
Signature

Header
{
"alg":"HS256",
"typ":"JWT"
}

payload:
{
"sub":"user123",
"role":"ADMIN",
"exp" : 1710000
}

Signature :
Signature ensures token is not tampered

JWT Authentication Flow (End-to-End)
1) Login request
Client send credentials ie username and password

2) Authentication by Spring Security
AuthenticationManager validates credentials

3)Generate JWT Token
If valid -> server generates JWT
String token = Jwts.Builder()
               .setSubject(username)
               .setExpiration(...)
                .signWith(secretKey)
               .compact();

4)Return Token to Client

5) Client Stores Token
Stores token in LocalStorage/Session/Cookies

6) Client sends token ineach request
Authoration : Bearer <JWT>

7)JWT Filter Validates Token
Extract Token
Validate Token
Check Expiry
Get USer Details

8) Set Authentication in Context
SecurityContextHolder.getContext()setAuthentication(auth);

9) Request proceeds to Controller

Core Components in Spring Boot JWT
1) JWT Utility Class
@Component
public class JWTUtil{
  private String secretKey = "secret";
  public String generateToken(String userName){
    return Jwts.Builder()
       .setSubject(username)
       .setIssuedAt(new Date())
      .setExpiration(new Date(System.currentTimeMillis()+1000 * 60 *15))
      .signWith(Keys.hmacShaKeyFor(SECRET_KEY.getBytes()),SignatureAlgorithm.HS256).compact();
  }

  public String extractUsername(String token){
    return Jwts.parseBuilder()
    .setSigningKey(secretKey.getBytes()).build()
    .parseClaims(token).getBody().getSubject();

  }
}

OAuth2 
Oauth2 is an authroziation framework that allows :
Application to access resources on behalf of a user
Without sharing user credentials
Instead of passwords, it uses access tokens

How OAuth2 Works
OAuth2 Client - Login
ResourceServer - API Protection
Authorization Server - Token Generation 
Step By Step
1) Client tries to access protected Endpoint
Get /api/user

2) Redirect to Authrozation Server
Spring Security automatically redirects
/outh2/authorization/google
User is sent to Googlr Login Page

3) User Authentication
User enters credentials
Grants permission

4) Authorization Server returns Code
http://localhost:8080/login/oauth2/code/google?code=abc123

5) Spring boot exchanges code for Token
/token
grant_type = authorization_code

6) Access Token returned

7) Client calls API with Token
Authorization Bearer token

8) Resource Server validates Token
Signature check
Expiry Check
Scopes Validation

Access Granted
