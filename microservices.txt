How do Microservcies Communicate With Each Other : 
There are 2 Different Types of Inter service Communication between microservices
1) Synchronous
2) Asynchronous 

Synchronous Communication :
The clients service waits for the response within a time limit. The possible solution is using Http protocol using via REST API for interservice communication.
Eg
REST API
GraphQl
Feign Using Eureka Discoveries
GRPC( 10 times faster than REST API)

Asynchronous Communication : The client Service doesnt wait for the response from another service. When the client microservice calls another microservice , thread is not blocked till the response comes from ther seerver.

The message producer service generates a message and sends the message to a message broker on a defined topic.
The message broker waits only for the acknowledgement from the message broker to know that the message is received by the broker.
The consuming service subscribes to the topic in the emssaging queue.All the messages belonging to the topic will be fed to the consuming system.
The message producer service and message consumer service dont even know each other.

In async communication,to inititate such type of communication a microservice who wants to send some data to another microservice publishes a message to a separate component known as message broker.
It is responsible for handling the message sent by the producer service and it will gurantee message delivery.

After the message is received by the broker, its now its job to pass the message to the target service.
If the recipient is down at the moment,the broker might be configured to retry a slong as necessary for sucessful delivery

These messages can be persisted if required or stored only in memory.In the latter case, they will be lost when the broker is restarted and they are not yet sent to the consumer.

Different messaging tools are based on AMQP(Advances Messaging Queueing Protocol)
Apache Kafka
RabbitMQ
ActivemQ

What if the Message Broker is Down
A message broker is a vital part of asynchronous architecture and hence must be fault tolerant.
This can be achieved by setting up additional standby replicas that can do failover.

 Its essential to ensure the messages arrive at its destination, a broker might be configured to work in at least once mode. Afte the message reaches the consumer,it needs to send back ACK to the broker. If no acknowledgement gets to the broker, it will retry the delivery after some time.

Types of Async Communication
1) Point to Point
2) Publisher Subscriber

PTP Async Communication :
A Queue will be used for this type of message-based communication.
The service that produces the message which is called as producer will send the message to queue in one messsage broker and the service that has an interest in that message which is called consumer will consume that message from queue and carry out further process for that message.

One messsage sent by the producer can be consumed by only one receiver and the message will be deleted after consumed.

If the receiver or an interested service is down, the message will remain persistent in that queue until the receiver is up and consumes that message.

For this reason, messaging based communication is one of the best choices to make our microservices resilient.

A popular choice for the queueing system is RabbitMQ, ActiveMQ

Publisher-Subscriber Async Communication :
In publisher-subscriber messaging-based communication the topic in the message broker will be used to store the message sent by the publisher and then the subsribers that subscribe to that topic will consume that message.

The message will be ready to consume for all subscribers and the topic can have one or more subscribers.The message remains persistent in a topic until we delete it.

In messaging-based communication the services that consume message either from queue or topic must know the common message structure that is prodcued or published by proudcer or subscriber.

eg are Kafka, Amazon SNS etc

Event Based Async Communication:
Unlike messaging-based communication, in event-based communication, especially in event-driven pattern, the servcies that consume the emssage do not need to know the details of the message.

In event-driven pattern services just push the event to the topic in message broker and then the services that subscribe to that topic will react for each occurence event in that topic. Each event in the topic will be related to specific business logic execution 

Pros and Cons of Synchronous Communication
Pros : Synchronous communication is simpler in design
Cons : Carries the risk of spreading failures across services.

Ways to mitigate Synchronous Communication

Pros and Cons of Asynchronous Communication :
Pros : Resilience and Scalability

       Loosely coupled microservices

KAFKA
Apache kafka is publish-subscribe based fault tolernace messaging system. It is fast, scalable and distributed by design

Pros of Kafka
1) Loose coupling : Neither service knows about each other regarding data update matters
2) Durability : Guarantees that the message will be delivered even if the consumer service is down. Whenever the consumer gets up again, all messages will be there
3) Scalibility : Since the messages gets stored in a bucket there is no need to wait for responses.
4) Flexibility : The sender of a message has no idea who is going to consume it.Meaning we can easily add new consumers with less work

Cons of Kafka
1)Semantics : The developers need to have deep understanding of the message flow as its strict requirements.
2) Message visibility : You must track all those messages to allow you to debug whenever a problem occurs.
